# 로드맵 & 개선 백로그

현재 `cpn_slab.html`은 **개념 검증용 시각화 프로토타입**입니다.
아래는 알려진 한계와 향후 개선 방향을 우선순위별로 정리한 백로그입니다.
작업 시 이 문서를 갱신해 주세요.

---

## 알려진 한계 (현재 상태)

| # | 한계 | 상태 |
|---|------|------|
| L1 | 순수 확률 기반 이동 | ✅ 해소 — 체류시간(proc)·용량(cap) 기반 결정적 엔진(`step()`)으로 교체 |
| L2 | 이동 발화가 규칙으로 강제되지 않음 | 🟡 부분 — 용량 조건으로 발화 강제. Guard 표현식의 데이터 정식화는 P1 잔여 |
| L3 | 강종 개수 하드코딩 | ✅ 해소 — `GRADES.length` 사용 |
| L4 | 자동화 테스트 없음 | 🟡 부분 — Playwright 헤드리스 스모크로 수동 검증. 저장소 내 테스트 하네스는 P3 |
| L5 | 실공정 데이터(MES) 연동 없음 | ⬜ 미해결 — 수동 투입까지는 구현(P4 잔여: 외부 소스 어댑터) |
| L6 | 상태가 메모리에만 존재 | ⬜ 미해결 (P5) |
| L7 | 반응형 레이아웃 미흡 | 🟡 부분 — KPI·차트·입력부는 auto-fit 그리드. SVG 다이어그램 스케일링은 P6 잔여 |
| L8 | 온도가 상태가 아닌 재계산값 | ⬜ 미해결 |

## 개선 백로그

### 우선순위: 높음

- **[P1] CPN 시맨틱 정식화** — 🟡 부분 완료
  ✅ 체류시간·용량 기반 발화 엔진(`step()`)으로 확률 이동을 대체.
  ⬜ 잔여: Place 마킹/Arc/Guard **표현식을 데이터 구조로 정식화**하고
  "발화 가능한 Transition 탐색 → 발화" 형태로 일반화.

- **[P2] 하드코딩 제거 / CONFIG화** — ✅ 완료
  매직넘버를 `CONFIG` 객체로 추출하고 입력부 UI와 연동. 강종 개수 하드코딩 제거.

- **[P3] 기본 검증 자동화** — 🟡 부분
  Playwright 헤드리스로 "시작→투입→완료→초기화" 스모크를 **수동 실행**해 검증 중.
  ⬜ 잔여: 저장소 내 `tests/` 하네스 + npm script 또는 CI로 상시화.

### 우선순위: 중간

- **[P4] 실데이터 연동 인터페이스** — 🟡 부분
  ✅ 수동 투입 대기열(`manualQueue`) + 입력부 UI로 사용자 주도 투입 구현.
  ⬜ 잔여: Heat 스케줄/설비 상태를 외부 소스(JSON/WebSocket/MES API)에서
  주입하는 어댑터 계층, 시뮬레이션 ↔ 실데이터 모드 토글. (L5)

- **[P5] 상태 영속화**
  진행 상태를 `localStorage` 또는 URL 파라미터로 저장/복원. (L6)

- **[P6] 반응형/모바일 레이아웃** — 🟡 부분
  ✅ KPI·심층분석·입력부는 auto-fit 그리드로 대응.
  ⬜ 잔여: SVG 다이어그램 `viewBox` 스케일링/재배치. (L7)

- **[P7] KPI 확장** — 🟡 부분
  ✅ 처리량, 강종별 리드타임, Place별 점유/용량, WIP 추이 추가.
  ⬜ 잔여: Place별 평균 대기시간, 설비 가동률(utilization), 강종 믹스 등.

### 우선순위: 낮음 / 장기

- **[P8] 코드 구조 분리**
  프로토타입이 커지면 단일 HTML을 `index.html` + `model.js` + `sim.js` +
  `render.js` 등으로 분리 (의존성 0 원칙은 유지, 번들러 도입은 별도 결정).

- **[P9] 시나리오 저장/재생**
  특정 투입 시퀀스를 저장해 재현 가능한 데모/디버깅 시나리오 구성.

- **[P10] 접근성**
  키보드 조작, ARIA 레이블, 색맹 대응(강종 구분을 색+패턴 병행).

## 설계 원칙 (개선 시 유지)
