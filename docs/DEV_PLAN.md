## 1. 목표 비전 (End-state)

1. **가시화**: 과거 데이터 기반으로 물류 흐름 재현·시각화
2. **병목 탐지**: 흐름에서 병목 지점 식별
3. **원인 분석**: 병목이 어떤 원인으로 발생했는지 진단
4. **개선안 도출**: 분석 결과로 개선 방안 도출
5. **라우팅 가이던스**: 제약을 반영하고 향후 물류 흐름을 고려해, **신규 슬래브 투입 시
   이동 방향을 추천**하는 의사결정 지원 시스템

## 2. 타당성 검토 (결론: **타당함**)

이 방향은 데이터 분석 성숙도 모델의 정석 경로를 그대로 따릅니다.

| 단계 | 분석 유형 | 질문 | 우리 자산 |
|------|-----------|------|-----------|
| 1 가시화 | Descriptive | "무슨 일이 있었나?" | 시뮬 엔진·차트 |
| 2 병목 탐지 | Descriptive/Diagnostic | "어디가 막혔나?" | 블로킹 기반 병목 탐지 |
| 3 원인 분석 | Diagnostic | "왜 막혔나?" | (신규) |
| 4 개선안 | Prescriptive | "무엇을 바꾸면 되나?" | 시뮬(디지털트윈)·최적화 스캐폴드 |
| 5 가이던스 | Decision support | "지금 이 슬래브는 어디로?" | 최적화 예시·제약 스펙 |

각 단계가 앞 단계 산출물 위에 자연스럽게 쌓이고, 지금까지 만든 모델·시뮬레이션·최적화
자산을 그대로 재사용하므로 **점진적·검증 가능한 확장**이 가능합니다.

### 강점
- 단계별로 **독립적 가치**를 제공(가시화만으로도 유용) → 리스크 분산
- 시뮬레이션 엔진이 이미 있어 **디지털 트윈 기반 what-if**로 곧장 확장 가능
- 최적화·제약 구조를 미리 스캐폴딩해 둠

### 반드시 짚어야 할 전제·리스크
1. **데이터가 최대 관건(크리티컬 패스).** 1~3단계는 과거 **이벤트 로그 품질**에
   전적으로 의존합니다. 필요한 최소 데이터: heat/slab별 **각 공정 진입·이탈
   타임스탬프**, 설비 상태, **지연/사유 코드**, 강종·폭·두께, 납기. → 첫 실무 과제는
   "무슨 데이터가 어떤 입도(granularity)로 있는가"를 파악하는 **데이터 감사**.
2. **모델 캘리브레이션·검증 필수.** 현재 `proc/cap`은 가정값입니다. 병목·개선안을
   신뢰하려면 실데이터로 처리시간·용량·변동성을 추정하고, **시뮬 KPI가 실적과
   일치하는지 검증**해야 합니다(그 전까지 4~5단계 결과는 참고용).
3. **병목은 고정점이 아니다.** 제품 믹스에 따라 병목이 **시간대별로 이동**합니다.
   정적 1점이 아니라 동적·시계열 병목 탐지가 필요.
4. **원인 분석엔 사유·상태 데이터가 필요.** 타임스탬프만으로는 부족 —
   설비 다운 사유, **blocking(하류 막힘) vs starving(상류 공급부족)** 구분,
   전환(강종/폭) 세트업 등. 필요 시 프로세스 마이닝 기법 활용.
5. **가이던스는 '자동 제어'가 아니라 '의사결정 지원'으로 시작.** 추천 + **근거 + 예상
   영향(리드타임/병목 변화)** 을 사람에게 제시하는 human-in-the-loop로 출발해야
   신뢰·안전·통합 리스크를 관리할 수 있음. 실시간 MES/L2 연동·폐루프 자동화는
   별도의 무거운 트랙.
6. **불확실성 반영.** 향후 흐름은 확률적이므로, 가이던스는 결정론적 단일 예측보다
   시나리오/강건(robust) 관점을 고려.
7. **현업 수용성(change management).** 추천을 **설명 가능**하게 만들어야 실제로
   쓰입니다("왜 이 경로인가").
